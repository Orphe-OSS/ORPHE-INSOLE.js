TRANSPORT AND STATE INSPECTION

Inspect acquisition paths, packet schemas, and measurement boundaries

This harness exercises three BLE data paths: SENSOR_VALUES notifications for Realtime, request–response polling of the device-side FIFO for buffered Raw acquisition, and STEP_ANALYSIS notifications for firmware-derived gait events. The UI calls the same Toolkit profile and measurement APIs intended for application code.

Transport integrity and biomechanical validity are separate concerns. Automated checks on this page validate delivery and state transitions only.
ACQUISITION PATHS

Three independent delivery contracts

Selecting a card applies its reference profile to every connected session. Use the six profile cards below for exact stream schemas.

TRANSPORT CONTRACTS

Delivery, continuity, and concurrency

Sample rate alone does not describe transport behavior. Compare delivery direction, buffering, recovery, and characteristic concurrency.

Realtime / SENSOR_VALUES Notification

The GATT characteristic pushes approximately 50 packets/s. Each packet contains two or four samples, depending on streamingMode.

Delivery
Peripheral → Host, continuous BLE Notification
Continuity
Serial gaps are observable but cannot be retransmitted
FIFO / Request–Response Polling

The Toolkit reads batches from the device-side ring buffer and tracks device serials. Host delivery is bursty and can lag acquisition.

Delivery
Host request → Peripheral response, batched samples
Continuity
Validate the final checkpoint after drain; exclusive with STEP_ANALYSIS
STEP_ANALYSIS Notification

Firmware emits motion packets plus overview, stride, and pronation packets. The Toolkit joins matching step numbers into completed rows.

Delivery
Peripheral → Host, separate GATT Notification stream
Concurrency
Can run with Realtime; shares BLE bandwidth; cannot run with FIFO
streamingMode selects the SENSOR_VALUES packet schema. 1 → header 50 / acc + gyro + quat / 200 sample/s / no press 3 → header 55 / acc + gyro + press / 200 sample/s / no quat 4 → header 56 / acc + gyro + press + quat / 100 sample/s

THE SAME API USED BY THIS PAGE

Minimal application integration

stopMeasurement() resolves after FIFO drain. The result contains only the formal measurement window.

buildInsoleToolkit(document.querySelector('#toolkit'), 'INSOLE 01', 0);
const session = getInsoleToolkitSession(0);
// After the user selects a device through the Bluetooth chooser:
await session.applyProfile('realtime-full');
await session.startMeasurement({ metadata: { subject: 'P001' } });
const result = await session.stopMeasurement();
const csv = insoleToolkitMeasurementToCSV(result, 'raw');
