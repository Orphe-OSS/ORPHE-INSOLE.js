# クォータニオン・スケール修正 検証レポート

対象: ORPHE-INSOLE.js v1.2.1

## 結論と根拠

- 本PRでは SENSOR_VALUES header 50 / 56 の quaternion を signed Q14（`1.0 = 16384`）としてデコードする。
- acc / gyro は従来どおり signed Q15（`1.0 = 32768`）であり、今回の変更対象ではない。
- Euler 角は quaternion を単位ノルムへ正規化してから計算する。
- 社内のBLEペイロード表では header 50 のバイト配置とFW世代の対応は確認できたが、Q14/Q15の除数は明記されていなかった。Q14判断は、NEC実測値と同系統の ORPHE-CORE.js における既修正実装の一致に基づく。

依頼時に共有されたNEC CSV解析結果（87,316サンプル、左右2台、約443秒）では、全サンプルの quaternion ノルムが約0.500（分位点0.49994〜0.50000）だった。旧SDKの `/32768` デコード値と一致し、行ごとに正規化すると yaw の約±20°の圧縮が解消して約±180°の範囲が復元した。

> マージ前ゲート: FW担当者に `1.0 = 16384` を明示確認し、FWバージョンによるスケール差がないことを確認する。現行の社内BLEペイロード表には除数の記載がないため、この確認が終わるまで本PRはドラフトのままとする。

## 既存CSVの救済手順（NEC共有用）

旧SDKで保存された quaternion は4成分が同じ比率で小さくなっているため、データは廃棄不要です。各行の `quat_w, quat_x, quat_y, quat_z` をその行のノルムで割り、長さ1へ正規化してください。ノルムが約0.5なら4成分を一律2倍することと同じです。その後、`euler_pitch, euler_roll, euler_yaw` は補正後の quaternion から再計算してください。Euler列そのものを2倍してはいけません。ノルム0または非有限値の行は無効として除外してください。
